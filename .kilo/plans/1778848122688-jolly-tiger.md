## 1. Phase 1 — International Students

### 1.1 Add `CountryOfOrigin` & Region Classification
**Why:** Needed for demographics reporting, regional quota management, and potential future regional admission rules.
- **Entity** `AdmissionApplication.cs:52` — add `string? CountryOfOrigin { get; set; }` (after `Nationality`)
- **Contract** `AdmissionContracts.cs:51` — add `string? CountryOfOrigin = null`
- **Migration** — `AddColumn` nullable `nvarchar(100)` `CountryOfOrigin`
- **Validation** — in `ValidateApplicantSpecificRequirements` (line 242), when `ApplicantType.International`, populate `CountryOfOrigin` from `PreviousInstitutionCountry` if missing
- **SaveApplicationRequest** (line 101) — add `public string? CountryOfOrigin { get; set; }`

### 1.2 English Proficiency Score Normalisation & Retrieval
**Why:** IELTS scores are ~9.0 scale, TOEFL is 0–120, PTE is 10–90. Currently stored/compared as raw strings with hardcoded thresholds (line 892–894). A scale-normalised score enables fair ranking across test types.
- **Entity** — add `decimal? EnglishProficiencyNormalizedScore { get; set; }` (computed on save)
- **Service** — extract `NormalizeEnglishProficiencyScore` private method; call from `SaveApplicationAsync` (line 76) before persisting
- **AutoAdmitAsync** (line 871) — use `EnglishProficiencyNormalizedScore` instead of raw string parsing
- **Config** — add `EnglishProficiency:MinimumScores` section to `appsettings.json`: `TOEFL: 80`, `IELTS: 6.5`, `PTE: 50`

### 1.3 Country-Specific Minimum Score Overrides
**Why:** Some countries (e.g., non-English-speaking) may require higher English thresholds.
- **Entity** — new `AdmissionInternationalSettings` table: `Id`, `CountryCode`, `MinTOEFL`, `MinIELTS`, `MinPTE`, `IsActive`
- **DbSet** in `LmsDbContext` — `DbSet<AdmissionInternationalSettings> InternationalSettings`
- **Migration** — create table with seed data for top 10 source countries
- **Service** — `ValidateEnglishProficiency` (line 364) accepts optional country override; `AutoAdmitAsync` passes `app.CountryOfOrigin`

### 1.4 Visa & Financial Proof Document Flag Wiring
**Why:** `VisaRequired`, `VisaApplicationNumber`, `FinancialProofProvided` already exist but no document types are `InternationalOnly` with `"visa"`/`"financial"` codes by default.
- **Data seeding** — add two `DocumentType` entries: Code=`VISA_LETTER`, Code=`FINANCIAL_PROOF`, both `InternationalOnly=true`, `IsCompulsory=true`, `Category=Admission`
- **Migration** — idempotent seed script (not EF migration) to insert if not exists
- **Service** — `GetRequiredDocumentsAsync` line 229 already handles `InternationalOnly`; no code change needed once seed data exists

## 2. Phase 2 — Direct Entry Students

### 2.1 Expand Supported Qualification Types
**Why:** Only A-Level, IJMB, NAPLEX recognised (line 333–335). HND, ND, Diploma, BTEC, IB Diploma, and Cambridge Advanced are common internationally.

**New enum** `DirectEntryQualificationType`:
```csharp
public enum DirectEntryQualificationType
{
    ALevel = 1,
    IJMB = 2,
    NAPLEX = 3,
    HND = 4,
    ND = 5,
    Diploma = 6,
    BTEC = 7,
    IBDiploma = 8,
    CambridgeAdvanced = 9
}
```

- **New entity** `AdmissionDirectEntryQualification`:
  | Field | Type |
    |---|---|
  | `Id` | `Guid` PK |
  | `AdmissionApplicationId` | `Guid` FK |
  | `QualificationType` | `DirectEntryQualificationType` |
  | `InstitutionName` | `string` |
  | `InstitutionCountry` | `string` |
  | `Grade` | `string` |
  | `Points` | `decimal` |
  | `YearObtained` | `int?` |
  | `SubjectsJson` | `string` (JSON array of `{subject, grade}`) |

  > **Design rationale:** Split from the monolithic `QualificationsJson` blob to enable structured queries (e.g., "show all HND applicants"). Keep `QualificationsJson` as the source of truth; this new table is a denormalised, indexable projection populated during `SubmitApplicationAsync` or via a projection query.

  > **Alternative considered:** Full normalised schema with separate tables for O-Level subjects, DirectEntry qualifications, and UTME scores. Rejected because it triples the table count for incremental benefit. The `QualificationsJson` blob covers dynamic subject lists well for O-Level; the new single table covers the fixed six qualification fields for DirectEntry.

- **Migration** — create `AdmissionDirectEntryQualifications` table
- **Service** — `HasDirectEntryQualification` (line 326) updated to include all 9 types
- **AutoAdmitAsync** (line 958) — min-score threshold (currently hardcoded `10` at line 963) made configurable per qualification type in `appsettings.json`

### 2.2 Configurable A-Level Point Scale Per Qualification Type
**Why:** A-Level → A*=6, A=5 … differs from HND → First=4.5, 2:1=3.5; and BTEC → D*=6, D=5.
- **New entity** `AdmissionQualificationScale`:
  | Field | Type |
  |---|---|
  | `Id` | `Guid` PK |
  | `QualificationType` | `DirectEntryQualificationType` |
  | `Grade` | `string` |
  | `Points` | `decimal` |
  
- **Seed data** — A-Level (*A–E, U), IJMB (*A–E, F), HND (First, 2:1, 2:2, Third, Pass), BTEC (D*–P), Cambridge (A*–E)
- **Service** — `CalculateDirectEntryScoreAsync(appId)` private method; sums points from `AdmissionDirectEntryQualifications` table using the scale; replaces `ParseQualificationsJson` for score calculation
- **AutoAdmitAsync** (DirectEntry case) — calls `CalculateDirectEntryScoreAsync`

### 2.3 Starting Level Auto-Suggestion by Qualification Type
**Why:** Each qualification type maps differently into program levels (HND holders → 300L for related programs; IB Diploma → 100L or 200L).
- **New entity** `AdmissionQualificationStartingLevel`:
  | Field | Type |
  |---|---|
  | `Id` | `Guid` PK |
  | `QualificationType` | `DirectEntryQualificationType` |
  | `ProgramType` | `ProgramType` (Undergraduate/Postgraduate) |
  | `DefaultLevelOrder` | `int` (1-indexed) |

- **Service** — `SuggestStartingLevelAsync(Guid applicationId)` private method; looks up config table and proposes `AcademicLevel`; called from `ValidateApplicantSpecificRequirements` (line 271) when `StartingLevelId` is null
- **UI** — front-end shows suggested level with "Accept suggestion" / "Override" buttons (no backend change needed)

### 2.4 Prerequisite Subject Validation for Direct Entry
**Why:** Direct entry applicants who studied, e.g., Engineering must have Maths/Physics at A-Level.
- **Entity** `AcademicProgram` — extend with `RequiredDirectEntrySubjectsJson` (JSON array of `{qualificationType, subject, grade}`)
- **Migration** — `AddColumn` `RequiredDirectEntrySubjectsJson nvarchar(max) NULL`
- **Service** — `ValidateDirectEntryPrerequisiteSubjects(qualsJson, requiredJson)` private method; throws `InvalidOperationException` if missing; called from `SubmitApplicationAsync` (line 175) for `ApplicantType.DirectEntry`

## 3. Phase 3 — Transfer & Exchange Students

### 3.1 Add `Exchange` Applicant Type to Enum
**Why:** Exchange students have fundamentally different requirements: no degree transfer, temporary stay, home institution verification, no tuition fee deduction. Currently represented only as a boolean flag (`IsExchangeProgram`), which conflates short-term exchange with full credit transfer.
- **Enum change** `ApplicantType.cs`:
  ```csharp
  UTME = 1, DirectEntry = 2, Transfer = 3, International = 4, Exchange = 5
  ```
- **Migration** — update existing rows from `Transfer(3)` where `IsExchangeProgram=true` → `Exchange(5)`
- **Service** — new `case ApplicantType.Exchange` in:
  - `GetRequiredDocumentsAsync` (line 202) — include `ExchangeOnly` documents, exclude `TransferOnly` and `DirectEntryOnly`
  - `ValidateApplicantSpecificRequirements` (line 242) — validate `HomeInstitutionName`, `HomeInstitutionCountry`, `ExchangeDurationMonths`
  - `AutoAdmitAsync` (line 814) — auto-admit with default score (exchange students usually pre-approved)
- **Entity** — remove `IsExchangeProgram` bool (replaced by enum); keep backwards-compatible via shadow property for 1 release cycle
- **Contracts** — add `ExchangeApplicantType` to `SaveApplicationRequest` and document the enum values in frontend

### 3.2 Add Home Institution & Exchange Duration Fields
**Why:** Required to verify the sending institution and enforce temporary stay limits.
- **Entity** `AdmissionApplication.cs:54` — replace `IsExchangeProgram` with:
  | Field | Type | Notes |
  |---|---|---|
  | `HomeInstitutionName` | `string?` | Required for Exchange |
  | `HomeInstitutionCountry` | `string?` | Required for Exchange |
  | `ExchangeDurationMonths` | `int?` | Required for Exchange (e.g. 1 or 2 semesters) |
  | `ExchangeProgramCode` | `string?` | e.g. ERASMUS, IAU, CHEA |
  
- **Migration** — drop `IsExchangeProgram`; add 4 new columns
- **Service** — added to `ValidateApplicantSpecificRequirements` under new `Exchange` case
- **Contracts** — add to `SaveApplicationRequest` (line 101: add 3 new optional string/int fields)

### 3.3 Credit Transfer Mapping Framework
**Why:** Currently credits are stored as a plain integer (`CreditsEarned`). There is no way to map specific courses from the previous institution to LMS courses. This prevents the registrar from knowing equivalent courses and generating a transfer credit plan.
- **New entity** `AdmissionTransferCredit`:
  | Field | Type | Notes |
  |---|---|---|
  | `Id` | `Guid` PK |
  | `AdmissionApplicationId` | `Guid` FK |
  | `SourceCourseCode` | `string` | e.g. `MATH101` |
  | `SourceCourseName` | `string` | e.g. `Calculus I` |
  | `SourceInstitution` | `string` | e.g. `Harvard` |
  | `TargetCurriculumCourseId` | `Guid?` FK → `CurriculumCourse` | Null = pending mapper review |
  | `CreditsAwarded` | `decimal` | Defaults to source credits |
  | `GradeEquivalent` | `string` | e.g. `A` |
  | `Status` | `enum: Pending / Approved / Rejected` | |
  | `ReviewedBy` | `string?` | Registrar userId |
  | `ReviewedAt` | `DateTime?` | |

- **New entity** `AdmissionTransferCreditMappingRule`:
  | Field | Type | Notes |
  |---|---|---|
  | `Id` | `Guid` PK |
  | `ProgramId` | `Guid` FK → `AcademicProgram` |
  | `SourceInstitutionPattern` | `string` | Regex/template for source institution name |
  | `SourceCourseCodePattern` | `string` | Regex/template |
  | `TargetCurriculumCourseId` | `Guid` FK → `CurriculumCourse` |
  | `MaxCreditsToAward` | `decimal` | Cap for this mapping |

- **Migration** — create both tables
- **Contracts** — `AdmissionTransferCreditResponse` record; `SaveTransferCreditsRequest`
- **Endpoints** — `TransferCreditMappingEndpoint.cs` (POST: bulk create mapping rules); `GetTransferCreditsEndpoint.cs` (GET by appId)
- **Service** — new method `MapTransferCreditsAsync(Guid applicationId)` called after admission offer; generates credit transfer plan viewable by registrar

### 3.4 GPA Scale Conversion & Configurable Credit Cap
**Why:** "Rigid credit-to-level mapping (line 784) and no credit cap" identified in assessment.
- **Entity** `AdmissionTransferEligibilityConfig` (or extend `AcademicProgram`):
  | Field | Type |
  |---|---|
  | `Id` | `Guid` |
  | `ProgramId` | `Guid` |
  | `CGPA4ScaleMin` | `decimal` (default 2.5) |
  | `CGPA5ScaleMin` | `decimal` (default 3.0) |
  | `MinCreditsRequired` | `int` (default 30) |
  | `CreditsPerLevel` | `int` (default 30, replaces hardcoded 30 in line 784) |
  | `MaxTransferableCredits` | `int` (default 60, 50% rule) |
  | `CreditsCapPercentage` | `decimal` (default 50.0) |

- **Migration** — `AddColumn` to `AcademicPrograms` (or new table; new table preferred for per-session override)
- **Service** — `ValidateTransferEligibilityAsync` (line 718) reads from config table instead of hardcoded constants; reports `MaxTransferableCredits` in `TransferValidationResult`

### 3.5 Semester/Term Selection for Transfer Students
**Why:** Transfer students often need to specify which semester they want to join (e.g., January vs September intake).
- **Entity** `AdmissionApplication` — add `Guid? TargetSemesterId` (FK to `LevelSemesterConfig`)
- **Migration** — `AddColumn` nullable `TargetSemesterId`
- **Service** — `ValidateTransferEligibilityAsync` (line 784) considers `TargetSemester` when suggesting level
- **UI** — dropdown in transfer form showing available upcoming semesters

## 4. Cross-Cutting Enhancements

### 4.1 Configurable Grading Scale Registry
**Why:** Direct entry and international students bring diverse grading scales (UK A-Level, US GPA, Indian percentage, Chinese 100-point). Currently none are converted to the institution's internal scale.
- **New entity** `AdmissionGradingScale`:
  | Field | Type |
  |---|---|
  | `Id` | `Guid` PK |
  | `Name` | `string` — e.g. `Nigerian 5.0 Scale` |
  | `Country` | `string?` |
  | `ApplicantType` | `ApplicantType` |
  | `EntriesJson` | `string` — `[{"grade":"A","points":5.0},{"grade":"B","points":4.0}]` |
  
- **Seed data** — Nigerian 5.0, US 4.0, UK A-Level, Indian %, Chinese 100pt
- **Service** — `ConvertGradeAsync(string scaleId, string rawGrade)` helper; used in auto-admit scoring and registrar review

### 4.2 Multi-Currency Fee Support
**Why:** "Fee calculation uses local currency; international students may need different payment options."
- **Entity** `FeeTemplate` (already exists) — extend existing `FeeAssignment` with `Currency` column
- **Migration** — `AddColumn` `Currency nvarchar(3) NULL` to `FeeAssignments`; default = `NGN`
- **Service** — `CalculateProgramFeeAsync` (line 676) returns `(amount: decimal, currency: string)`; frontend displays both
- **Config** — add supported currencies to `appsettings.json`: `"NGN", "USD", "EUR", "GBP"` with exchange rates for display only (actual payment still in NGN at time of invoicing)
- **Contracts** — `FeeAmountResponse(decimal Amount, string Currency, decimal? EquivalentNgn)`

### 4.3 Home Institution Verification for Transfer & Exchange
**Why:** No mechanism exists to verify previous institution accreditation.
- **Entity** `AdmissionInstitutionVerification`:
  | Field | Type |
  |---|---|
  | `Id` | `Guid` PK |
  | `AdmissionApplicationId` | `Guid` FK |
  | `InstitutionName` | `string` |
  | `Country` | `string` |
  | `VerificationStatus` | `enum: Pending / Verified / NotAccredited / ManuallyOverridden` |
  | `VerifiedAt` | `DateTime?` |
  | `VerifiedBy` | `string?` |
  | `Notes` | `string?` |

- **Migration** — create table; seed with top 50 accredited universities in Nigeria and common sending countries
- **Endpoints** — `VerifyTransferInstitutionEndpoint.cs` (POST/DELETE context response); `GetTransferVerificationStatusEndpoint.cs`
- **Service** — `VerifyInstitutionAsync(string name, string country)`; flags non-accredited institutions before `SubmitApplicationAsync`

### 4.4 Code Quality Refactoring — Eliminate Duplication in `GetRequiredDocumentsAsync`
**Why:** The same filtering logic appears identically at lines 202 and 461. Single source of truth prevents divergence.
- **Service** — extract method `FilterDocumentsByApplicantType(IEnumerable<DocumentType> allDocs, ApplicantType type)`; call from both call sites
- **Risk:** None — pure extraction, identical output confirmed by existing auto-admit integration tests

### 4.5 Feature Flag for Gradual Rollout of Exchange Type
**Why:** Changing `ApplicantType` enum and updating existing `Transfer` rows with `IsExchangeProgram=true` is a data migration that must be reversible.
- **Config** — add `"Admission:FeatureFlags:ExchangeApplicantTypeEnabled": false` to `appsettings.json`
- **Service** — `GetRequiredDocumentsAsync` and `ValidateApplicantSpecificRequirements` guard new `Exchange` case behind feature flag
- **Endpoint** — `/admin/features/exchange-type` GET/POST to toggle
- **Migration** — conditional: created tables/columns are always applied; enum value and data migration only run when flag is ON

```
  └── 1.3  Country-specific minimum score overrides
  └── 2.3  Starting level auto-suggestion

```

## 7. Risks & Mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| Breaking change to `ApplicantType` enum shifts IDs | Medium | Used `[EnumMember]` attributes; data migration script runs before app start; feature flag gates new case until verified |
| `QualificationsJson` schema evolution breaks existing read paths | Low | Only additive changes (new property keys); backward-compatible `TryGetProperty` pattern already used throughout |
| Auto-admit score re-ranking causes unexpected admit changes | Low | Dry-run endpoint `/admin/auto-admit?isDryRun=true` already exists (line 814); run with `isDryRun=true` first every cycle |
| Credit transfer table grows without cleanup for rejected apps | Low | Background job (Phase 7) archives `TransferValidationResult.Reason` to `AdmissionDecisionAuditLog`; tracks FF foreign keys |
| Multi-currency fee calculation misleads applicants | Low | All amounts displayed in NGN as authoritative; foreign currency shown as display-only with prominent disclaimer |
| Exchange students admitted without quota controls | Medium | Exchange case in `AutoAdmitAsync` defaults to score = PreviousCGPA with min=0; respects `MaxAdmissions` quota |
| Migration runs on production without backup | Low | Every migration script wrapped in `IF NOT EXISTS` guards; peer-review gate before production deploy |

## 8. Out of Scope (Deferred to Phase 7 or Later)

| Item | Reason |
|---|---|
| WES/ECE credential evaluation API integration | External dependency, licensing cost, requires vendor contract |
| Visa / I-20 / DS-2019 automated generation | Legal/immigration compliance — requires legal review |

| Dynamic multi-currency payment processing (Stripe/PayPal in NGN, USD etc.) | Payment gateway contract renegotiation needed |
| Exchange student transcript evaluation | Covered by institution verification stub; full evaluation is phase 7 |
